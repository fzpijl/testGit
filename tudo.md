# 工具链相关工作计划

## vscode 插件相关

### 插件更新

目前插件并没有更新机制，如果插件功能有更改，我们需要打包一份新的插件，然后发送给插件使用者，由插件使用者自己手动安装新的插件，对于插件使用者来说，这种更新插件的方式很不友好。计划提供友好的插件更新机制。有三种提供插件更新的机制：

1. 使用vscode官方提供的更新功能，将新插件发布到官方仓库，即可获得更新功能。这种方式符合编辑器的使用者的习惯，也无需额外的开发成本，但由于非技术原因排除此种方式。

2. 开发一个插件，用于管理其他插件的更新。由于目前计划是将所有插件功能都集中到一个插件中，为了一个插件的更新再开发另一个插件没有太大必要，此时暂不考虑此方案，如果以后有开发多个插件的计划，可以采用此种方式。

3. 将插件的更新机制集成到插件中。当插件有更新时，将新版插件打包放到本地的静态服务器中，在一定的时机去服务器读取更新。

计划使用第3种方式。

### 应用打包速度优化

目前插件提供的应用模板使用 webpack 处理应用的打包工作，处理过程包含代码格式检查、转译TS/JS代码、代码压缩混淆、处理图片和模型等资源、拷贝其他静态资源、将整个应用打包成zip格式然后上传到设备。以 irisVRLauncher 项目为例，整个过程至少需要 6s，随着以后应用规模扩大，包含更多的代码和图片、模型等静态资源，处理时间也会变得更长，开发应用的时，每次有改动都需要等待一定的时间才能看到效果，对开发效率有一定的影响，所以计划优化打包过程花费的时间。目前有 2 个方向：

 1. 基于现有 [webpack](https://webpack.js.org/) 进行优化，优点是 webpack 工具比较成熟，我们需要功能基本都有现成的解决方案，而且是用 JS 语言开发，上手比较容易。

 2. 使用比较新的工具如 [esbuild](https://esbuild.github.io/)、 [turbo](https://turbo.build/pack) 、[swcpack](https://swc.rs/docs/usage/bundling)等，这些工具是采用 go 或 rust 语言开发，运行效率相较于使用 JS 开发的 webpack 要快很多，缺点是功能还不够完善，我们需要的功能不一定都有现成的解决方案。

具体采用哪种方案还需要进一步调研。

### 增强现有输出应用log功能

现有的log输出功能还比较简陋，已知有以下几点可以进行改进：

1. 通过不同进程过滤输出，现在的实现是将 `adb logcat` 命令拿到的数据都展示出来，其中包含系统和所有应用的输出，输出的信息量非常大，一般情况使用者并不关心这些无关信息，使用者只关心自己应用的输出。

2. 目前提供了通过字符串或是正则表达式过滤输出的功能，但是对于不同行之间数据有相关性的情况，例如错误栈信息，不能很好的展示。当过滤条件匹配错误栈中某一行时，应该将整个错误栈的信息都显示出来。

3. 基于性能考虑，目前默认存储5000行数据，这在数据较多时，会丢失较早时输出的数据，计划将这个限制提高一个较合理的值。

### 添加代码热更新机制

开发应用时，如果代码有改动，需要手动点击插件的启动按钮，重新执行打包上传，然后重启应用，加入热更新功能，当在编辑器中保存修改时，可以自动执行上述过程，方便开发者。此功能的开发大概会分为两个阶段：  
1. 代码更新时，会重启应用
2. 不会重启应用，动态更新变化的代码

## IDL项目

已经开发了部分功能，在开发过程中遇到很多在设计上需要做取舍的困难，项目存在一定的风险，由于暂时没有足够的资源分配到此项目，所以项目暂时处于停滞状态。

## XRWeb - JS SDK 封装

由于底层 UI 库的重写，所以 SDK 中与 UI 相关的部分也需要全部重写。之前计划通过 IDL 工具完成重写，但由于 IDL 项目的不确定性，重写工作也可能与之前一样通过手工完成。  
由于之前使用新 UI 库开发应用的需要，应用组已经封装了一部分 UI 库相关接口，已经满足开发应用的部分功能需求，如果开发应用有新的需求，我们会在此基础上继续封装。


## React 自定义渲染器

在老版 UI 库中，已经实现了 React 自定义渲染器，并使用 React 开发了资源浏览器应用。新版 UI 库的自定渲染器的工作还没有开始，因为自定义渲染器的实现严重依赖 XRWeb，在 XRWeb 有不确定性(IDL vs 手工)的情况下，还没有在自定义渲染器上投入时间。
